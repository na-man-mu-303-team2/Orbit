## Current Conclusion

The audience engagement feature has a working foundation, but the feature check found several plan gaps. This document turns those gaps into implementation tasks and records the decisions needed for implementation.

All product and technical planning questions identified during this planning pass have been resolved. Implementation can proceed from this document.

## Remaining Scope

### Must Complete

- Normalize audience-facing Korean error copy for join/session failures.
- Generate audience slide snapshots during session preparation and freeze them at session start.
- Render audience slide state with image-first behavior and Deck JSON fallback.
- Replace presenter setup placeholders with real controls for prepared interactions, ordering, survey status, and AI reference selection.
- Complete presenter live controls for ad hoc poll/quiz creation, activation, close, manual result exposure, and live results.
- Complete poll/quiz behavior for all planned question types, result visibility rules, quiz answer reveal, and `speed-bonus` scoring.
- Implement Q&A duplicate merge instead of one group per question.
- Implement grounded AI Q&A using public deck content and selected references only.
- Deliver private AI answers by WebSocket push, with REST recovery retained.
- Add regression tests and E2E coverage for the completed behavior.

### Out Of Scope For This Completion Pass

- Question upvotes.
- Audience-visible global question list.
- Presenter pointer, progress, or remaining-time sync.
- Legal approval of final contact consent wording.
- Fixing non-audience smoke failures recorded in the feature check.
- Changing unrelated editor, rehearsal, upload, or AI suggestion flows.

## Existing Constraints To Preserve

- Audience entry remains `/join` and `/join/:joinCode`.
- Audience login is not used.
- Audience identity uses required nickname and session-scoped signed `HttpOnly` cookie.
- All participation features start disabled.
- Disabled features are hidden from the audience UI while preserving already submitted data.
- Only one poll or quiz can be active at a time.
- Audience-facing payloads must never include speaker notes, raw transcript, raw audio, presenter script, file base64, tokens, cookies, passwords, or secrets.
- AI Q&A answers are visible only to the asker.
- Presenter queue statuses remain `pending` and `answered`.
- Survey CSV remains survey-only.

## Architecture Decisions Already Fixed

- Shared contracts in `packages/shared` remain the source of truth for request/response and WebSocket payload shapes.
- API remains under `/api/v1/presentation-sessions`.
- Realtime uses `AudienceRealtimeGateway` with session, presenter, and private audience rooms.
- Python worker handles AI Q&A through `/qna/answer`.
- Raw audience data retention remains 30 days, followed by anonymous aggregate retention.
- Default user-facing copy language is Korean.

## Confirmed Decisions

- Audience slide snapshots will be generated by a separate Node worker/package dedicated to slide rendering jobs.
- Deck JSON fallback must be full fidelity: it should match the editor/presenter slide renderer output, including animations/effects final state.
- `resultVisibility = manual` uses per-question exposure state so multi-question interactions can expose results independently.
- `quizScoring = speed-bonus` uses integer points from `1000` down to `500` for correct answers; wrong answers score `0`.
- `speed-bonus` requires per-question `timeLimitSeconds`.
- Quiz answer reveal after close shows the correct answer, the audience member's own submitted answer, and their correctness state.
- AI Q&A uses the configured OpenAI chat model when `OPENAI_API_KEY` is present; otherwise it returns a structured grounding failure and escalates to presenter.
- Duplicate Q&A merge is conservative and uses cosine similarity at or above `0.88`.
- The detailed presenter control/results screen will use `/presentations/:sessionId/audience`.
- Snapshot generation failure does not block session start; failed slides use the full Deck JSON fallback.
- `speed-bonus` uses the linear formula `score = round(500 + 500 * remainingTimeRatio)`, where `remainingTimeRatio = max(0, timeLimitSeconds - elapsedSeconds) / timeLimitSeconds`.
- `timeLimitSeconds` must be between `10` and `120` seconds when `quizScoring = speed-bonus`.
- AI grounding requires top chunk similarity at or above `0.78` and at least one selected source or public deck source citation.
- Slide rendering uses both `apps/slide-render-worker` and `packages/slide-renderer`; the app owns job processing and the package owns pure rendering logic.
- Slide snapshots are generated during session preparation and refreshed at session start when the deck version or content hash changed.
- Slide snapshots are stored through the existing `StoragePort` with the new file purpose `audience-slide-snapshot`.
- AI Q&A citations shown to the asker include source type and title only.

## Resolved Questions

These questions were answered on 2026-07-05.

### Q1. Audience Slide Snapshot Generation

**Decision:** Option B. Generate snapshots in server-side rendering tooling. Q9 later narrowed the owner to a separate Node worker/package dedicated to slide rendering jobs.

The approved plan requires image-first rendering with snapshots generated/frozen at session start. The implementation must use server-side rendering rather than relying on the web presenter client or existing thumbnails.

### Q2. Deck JSON Fallback Fidelity

**Decision:** Option B. Full fallback: render the same visual output as the editor/presenter slide renderer, including animations/effects final state.

When `slideSnapshotUrl` is missing or fails, the Deck JSON fallback must aim for presenter-renderer parity rather than a simplified or message-only fallback.

### Q3. Manual Result Exposure Model

**Decision:** Option B. Add per-question exposure state so multi-question interactions can expose results independently.

`resultVisibility = manual` remains hidden until the presenter exposes results for that specific question.

### Q4. Speed Bonus Formula

**Decision:** Option B plus Window A. Correct answers use integer points from `1000` down to `500` based on response speed; wrong answers score `0`. Per-question `timeLimitSeconds` is required when `quizScoring = speed-bonus`.

### Q5. Quiz Answer Reveal Content

**Decision:** Option B. After close, audience sees the correct answer, their own submitted answer, and correctness state.

### Q6. AI Q&A Answer Generation Strategy

**Decision:** Option A. Use the configured OpenAI chat model when `OPENAI_API_KEY` is present; otherwise return a structured grounding failure and escalate.

### Q7. Duplicate Q&A Merge Threshold

**Decision:** Option A. Conservative semantic merge with cosine similarity at or above `0.88`.

### Q8. Presenter Control Surface Route

**Decision:** Option B. Add a dedicated session route under the existing app router. Q13 later fixed the route as `/presentations/:sessionId/audience`.

### Q9. Server-Side Slide Snapshot Owner

**Decision:** Option C. Add a separate Node worker/package dedicated to slide rendering jobs.

This keeps React/Konva-compatible rendering in the TypeScript/Node side of the stack and avoids forcing slide rendering through the Python AI worker.

### Q10. Snapshot Failure Policy At Session Start

**Decision:** Option A. Start session anyway and use the full Deck JSON fallback for failed slides.

Snapshot generation failure must be visible to presenter/operator logs but must not prevent the live session from starting.

### Q11. Exact `speed-bonus` Formula

**Decision:** Option A. Linear formula: `score = round(500 + 500 * remainingTimeRatio)`, where `remainingTimeRatio = max(0, timeLimitSeconds - elapsedSeconds) / timeLimitSeconds`.

Wrong answers score `0`. Responses after the time limit are invalid for speed-bonus quizzes unless the interaction has already closed and rejects submissions globally.

### Q12. `timeLimitSeconds` Bounds

**Decision:** Option B. `10..120` seconds.

The shared schema and presenter UI controls must enforce the same range.

### Q13. Exact Presenter Control Route

**Decision:** Option A. `/presentations/:sessionId/audience`.

Navigation links, tests, and deep links should use this as the canonical presenter audience control/results route.

### Q14. AI Grounding Confidence Threshold

**Decision:** Option A. Conservative: top chunk similarity at or above `0.78`, with at least one selected source or public deck source cited.

Weak grounding below this threshold must return a structured failure and escalate to presenter instead of generating a global-knowledge answer.

### Q15. Slide Rendering Worker Shape

**Decision:** Option C. Add both `apps/slide-render-worker` and `packages/slide-renderer`, where the app owns job processing and the package owns pure rendering logic.

Use package names `@orbit/slide-render-worker` and `@orbit/slide-renderer`.

### Q16. Slide Snapshot Generation Timing

**Decision:** Option A. Generate during session preparation, then refresh at session start if the deck changed.

Deck change detection should use the persisted deck version when available and a content hash fallback when a version is not available.

### Q17. Slide Snapshot Storage

**Decision:** Option A. Use the existing `StoragePort` and create the new file purpose `audience-slide-snapshot`.

Snapshot assets are audience-facing generated assets, but they should not appear as ordinary user-managed reference material.

### Q18. AI Q&A Source Citation Requirement

**Decision:** Option A. Store and return source type plus title only. Supported source types are `deck-slide` and `reference-material`; titles are the public slide title or reference file title.

Do not show snippets or full source metadata to the audience in this completion pass.

## Risks And Mitigations

| Risk | Impact | Mitigation |
| --- | --- | --- |
| Snapshot generation can add runtime complexity or slow session start. | High | Use the dedicated Node worker/package, generate during preparation, and fall back to full Deck JSON without blocking session start. |
| Manual result visibility can conflict with existing aggregation assumptions. | Medium | Add explicit state and tests for every visibility enum. |
| AI answer quality can look confident without grounding. | High | Require source chunks and confidence threshold; escalate on weak grounding. |
| Duplicate merge can hide distinct questions. | Medium | Start with conservative threshold unless product chooses otherwise; preserve individual submissions under the group. |
| Presenter controls can grow too large for the current panel. | Medium | Use `/presentations/:sessionId/audience` for detailed controls and keep the live overlay focused on fast actions. |
| Python worker test tooling may be unavailable locally. | Medium | Restore `uv` workflow or document local setup before relying on Python tests. |
